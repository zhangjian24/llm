瀑布模型是软件工程中最经典的线性顺序开发模型。其核心思想是将软件生命周期划分为一系列固定的、顺序的阶段，每个阶段必须在前一阶段完全结束后才能开始，且每个阶段都有明确的目标、工作内容、输入和输出。

以下是瀑布模型各阶段的详细拆解：
| 阶段 | 子阶段 (如有) | 核心输入 (Input) | 核心活动 (Activity) | 核心输出 (Output/Deliverable) |
| :--- | :--- | :--- | :--- | :--- |
| 1. 计划 | - | 业务构想、市场调研 | 可行性分析、资源规划、风险评估 | 《项目计划书》 |
| 2. 需求 | - | 《项目计划书》 | 需求挖掘、业务流程分析、原型确认 | 《软件需求规格说明书》(SRS) |
| 3. 设计 | 3.1 概要设计 | SRS | 技术选型、架构模式定义、模块划分、部署规划 | 《系统架构说明书》(SAD) |
| | 3.2 数据设计 | SAD + SRS | 概念/逻辑/物理模型设计、ER图绘制、索引优化 | 《数据库设计说明书》(DBD) |
| | 3.3 详细设计 | SAD + DBD | 类/对象建模、时序图绘制、API接口定义、算法伪代码 | 《详细设计说明书》(DDD) <br> *(含API文档)* |
| 4. 编码 | - | DDD + DBD | 环境搭建、代码实现、单元测试、代码审查 | 源代码 <br> 《单元测试报告》 <br> 可执行文件/安装包 |
| 5. 测试 | - | 可执行文件 + SRS | 集成测试、系统测试、性能测试、用户验收测试(UAT) | 《测试分析报告》 <br> 合格软件产品 |
| 6. 维护 | - | 合格产品 + 用户反馈 | 缺陷修复、环境适配、功能增强、性能优化 | 软件补丁/新版本 <br> 《维护日志》 |
