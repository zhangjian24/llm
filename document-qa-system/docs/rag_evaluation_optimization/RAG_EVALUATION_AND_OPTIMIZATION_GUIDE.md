# RAG系统评估与优化指南

## 一、评估体系

### 1.1 评估维度概述

RAG评估分为两大层：**检索层**和**生成层**。研究表明，检索准确性只能解释60%的端到端RAG质量变异，其余40%取决于模型如何使用检索到的上下文。

### 1.2 核心指标体系

#### 生成质量指标

| 指标 | 衡量内容 | 需要参考答案 | 目标阈值 |
|------|---------|-------------|----------|
| **Faithfulness** | 答案中的每个陈述是否被检索上下文支持 | 否 | ≥0.8 |
| **Answer Relevance** | 生成的答案是否真正回答了问题 | 否 | ≥0.75 |
| **Hallucination Rate** | 包含未支持声明的响应比例 | 否 | <5% |

**Faithfulness解读**：
- 低分表示模型使用训练知识填补检索空白
- 需优化检索或约束system prompt
- 医疗/金融/法律场景应 ≥0.9

#### 检索质量指标

| 指标 | 衡量内容 | 需要参考答案 | 目标阈值 |
|------|---------|-------------|----------|
| **Context Precision** | top-K文档中相关块是否排在前面 | 否 | ≥0.7 |
| **Context Recall** | 检索上下文是否包含回答问题所需的全部信息 | **是** | ≥0.75 |
| **Precision@K** | 检索的K个文档中有多少相关 | 是 | ≥0.7 |
| **Recall@K** | 所有相关文档中有多少被检索到 | 是 | ≥0.8 |
| **MRR** | 首个相关文档的排名位置 | 是 | 越高越好 |
| **NDCG** | 同时考虑相关性和位置的打分 | 是 | ≥0.8 |

#### 关键洞察

- **Context Precision低** → 嵌入模型排序差 → 需加cross-encoder重排
- **Context Recall低** → 检索遗漏 → 增大chunk overlap或K值

### 1.3 评估框架对比

| 框架 | 最佳场景 | CI/CD集成 | 自托管支持 |
|------|---------|----------|------------|
| **Ragas** | 快速实验、标准指标、合成数据生成 | 需手动 | ✅ |
| **DeepEval** | CI/CD质量门、生产监控 | ✅原生pytest | ✅ |
| **TruLens** | 开发时A/B实验、仪表盘监控 | 不适合 | ✅ |

**推荐工作流**：用Ragas生成合成数据集 → 用DeepEval做CI/CD质量门 → 用TruLens/Langfuse做生产监控

### 1.4 评估数据集构建

1. **Golden数据集**（50-200条）：人工标注的高质量问答对，覆盖边界情况和多跳问题
2. **合成数据集**：用LLM从文档生成，快速但需人工审核
3. **生产回填数据集**：从真实用户query中采样标注，最具代表性

### 1.5 生产监控指标

| 指标 | 告警阈值 | 排查方向 |
|------|---------|----------|
| Faithfulness采样 | < 0.75 | 最近文档摄入质量 |
| Answer Relevance | < 0.70 | 查询分布偏移 |
| Hallucination率 | > 5% | 新查询类型检索覆盖 |
| P95检索延迟 | > 500ms | 索引大小、embedding模型负载 |
| 上下文利用率 | < 40% | chunk大小、重排过滤效果 |

## 二、优化策略体系

### 2.1 实施优先级

```
第一层（基础）: 混合搜索 → Chunking优化 → HNSW调优
第二层（模型）: Embedding微调 → LLM微调
第三层（高级）: 查询转换 → 重排 → 语义缓存 → Agentic RAG
```

### 2.2 索引层优化

#### 1. Chunking策略优化

| 策略 | 适用场景 | 效果 |
|------|---------|------|
| 固定长度 | 简单文档 | 简单但可能破坏语义边界 |
| 语义分块 | 复杂结构文档 | 按段落/标题分割，保持语义完整 |
| 多表示索引 | 大型文档库 | 同时索引摘要、段落、元数据 |

**关键原则**：每个chunk应代表一个完整思想，而非半个想法

#### 2. 多表示索引 (Multi-Representation)

为同一文档存储多种语义表示：
- 摘要版本
- 段落级chunks
- 元数据描述
- 提取的实体摘要
- 图表caption

### 2.3 检索层优化

#### 3. 混合搜索 (Hybrid Search)

结合 **BM25关键词搜索** + **向量语义搜索**

- BM25：擅长精确匹配（零件号、术语、法规代码）
- 向量搜索：擅长语义理解

**效果**：检索recall提升3-3.5倍，端到端准确率提升11-15%

#### 4. HNSW索引调优

| 参数 | 作用 | 建议 |
|------|------|------|
| M | 每节点最大连接数 | 提高→更好recall，更多内存 |
| EF_CONSTRUCTION | 建索引时搜索深度 | 提高→更好的图结构，更长建索引时间 |
| EF_RUNTIME | 查询时搜索深度 | **最关键**→可动态调整 |

#### 5. 查询转换 (Query Transforms)

| 技术 | 原理 | 适用场景 |
|------|------|---------|
| **HyDE** | 用LLM生成假设答案，再用假设答案检索 | 模糊查询、稀疏数据集 |
| **多查询生成** | 并行生成多个查询改写版本 | 歧义性查询 |

#### 6. 重排序 (Re-ranking)

两阶段检索：
1. **召回阶段**：向量/混合搜索获取候选集(top 20-100)
2. **精排阶段**：Cross-encoder模型精细排序

效果：精确度提升10-40%

### 2.4 模型层优化

#### 7. Embedding微调

通用embedding模型在专业领域表现差，需用对比学习或MNR Loss微调。

#### 8. LLM微调

使用LoRA等技术适配：
- 特定回复格式
- 合规语言风格
- 引用方式

### 2.5 高级架构模式

#### 9. Graph RAG

将知识图谱融入检索：
- 节点=实体/概念
- 边=关系

**优势**：支持多跳推理

#### 10. RAPTOR分层检索

构建文档摘要树结构：
- 叶子节点→细粒度信息
- 中间节点→上下文摘要
- 根节点→高层概念洞察

#### 11. Agentic RAG

AI Agent动态编排管道：
- 判断何时需要检索
- 重新构造查询
- 调用外部工具
- 多步推理
- 验证答案

#### 12. 上下文压缩

在检索和生成之间增加压缩层：
- 提取与查询相关的片段
- 减少token使用
- 降低幻觉

### 2.6 一致性与成本优化

#### 13. 语义缓存

相似查询返回缓存结果：
- 相同问题→相同答案（一致性）
- 降低延迟（ms vs 秒）
- 成本降低可达73%

#### 14. 长期记忆管理

多轮对话中：
- 存储交互历史为向量
- 检索与当前问题相关的历史
- 避免上下文膨胀

## 三、成本优化

### 3.1 成本构成

| 成本项 | 占比 | 优化优先级 |
|--------|------|------------|
| LLM生成 | 60-80% | ★★★★★ |
| Embedding API | 15-25% | ★★★ |
| 向量数据库 | 5-15% | ★★★ |
| 基础设施 | 5-10% | ★★ |

### 3.2 Embedding成本优化

#### 1. 批量处理

- **批量API调用**：将多个文本一次性嵌入
- **批量大小**：100-500条/批
- **异步处理**：并行处理

**效果**：降低30-40%

#### 2. 查询Embedding缓存

| 缓存类型 | 原理 | 命中率 |
|----------|------|--------|
| 精确匹配 | 哈希key存储 | 10-20% |
| 语义缓存 | 向量相似度匹配 | 30-50% |

#### 3. 模型选择

- 小模型可能与大型号效果相当：BGE、E5、all-MiniLM
- 领域微调可匹配大型号性能

### 3.3 LLM成本控制（最大优化项）

#### 1. 上下文长度管理

| 策略 | 效果 |
|------|------|
| 检索更少但更相关的chunks | 40-60%成本降低 |
| 动态上下文大小 | 按查询复杂度调整 |
| 上下文压缩 | 总结后再注入 |
| 相关性阈值过滤 | 低于阈值直接丢弃 |

#### 2. 输出控制

```python
# 指定输出长度
"Respond in 2-3 sentences maximum."
"List only the top 3 recommendations."
"Answer in under 50 words."
```

#### 3. 模型路由

```
60% 简单查询 → GPT-3.5/Haiku ($0.50/1M)
30% 中等查询 → GPT-4o-mini/Sonnet ($5-15/1M)  
10% 复杂查询 → GPT-4o/Claude Opus ($30-75/1M)
```

**效果**：整体成本降低50-70%

#### 4. Prompt缓存

| 提供商 | 缓存写入 | 缓存命中 | 节省 |
|--------|----------|----------|------|
| Anthropic | 1.25x | 0.1x | 90% |
| OpenAI | 自动>1024tokens | 50%折扣 | 50% |

#### 5. 响应缓存

| 类型 | 适用场景 | 命中率 |
|------|---------|--------|
| 精确匹配 | 相同问题 | 10-20% |
| 语义缓存 | 相近问题 | 61-68% |

### 3.4 向量数据库成本优化

| 技术 | 效果 |
|------|------|
| 量化(PQ) | 存储降低4-8x |
| 维度reduction | 1536→768维，存储减半 |
| HNSW调优 | 平衡延迟与成本 |

### 3.5 五大成本控制策略总结

| 策略 | 节省比例 | 实施难度 |
|------|---------|----------|
| Prompt优化 | 20-40% | 小时级 |
| 响应缓存 | 30-70% | 低 |
| 模型路由 | 40-60% | 中 |
| Prompt缓存 | 50-90% | 低 |
| 自托管 | 60-90% | 月级 |

## 六、关键成功因素

1. **先评估后优化**：用RAGAS等框架建立基线指标
2. **先基础后高级**：检索优化优先于生成优化
3. **持续监控**：质量指标+成本指标并行追踪
4. **按需选择**：根据场景选择技术组合，非全部采用
5. **A/B测试**：重大优化需验证后再全量

---

## 七、常见误区

1. **用同一模型生成和评估** → 分数虚高，用不同模型做judge
2. **只看端到端指标** → 分离检索/生成指标才能定位问题
3. **阈值设置过高** → 初期建议0.7，逐步收紧
4. **合成数据不审核** → LLMs可能生成看似合理但实际错误的内容
5. **跳过per-component评估** → 端到端正确性只能告知问题所在，无法定位哪一层出错

## 八、实践建议

### 8.1 评估建议

- 50条golden questions足够稳定跟踪指标变化
- 用GPT-4o-mini做judge降低成本，关键场景用GPT-4o
- 每次知识库大更新后重新跑完整评估
- 评估数据集需版本控制，与代码一起管理
