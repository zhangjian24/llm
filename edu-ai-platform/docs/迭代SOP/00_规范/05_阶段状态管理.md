# 阶段状态管理

> 文档标识：SOP-STM-001
> 版本：1.0
> 更新日期：2026-03-31
> 维护人：SOP管理员
> 状态：已发布

> 本文档定义AI数字员工在迭代执行过程中如何管理上下文、阶段状态和任务状态。
> 确保AI能够跟踪迭代进度、维护必要的上下文信息，并在活动间传递信息。

---

## 1. 状态管理概述

### 1.1 状态管理目标

| 目标 | 说明 |
|------|------|
| 上下文连续 | 活动执行时能够获取必要的上下文信息 |
| 进度可见 | 阶段和活动的进度对人类可见 |
| 依赖追踪 | 跟踪任务间的依赖关系 |
| 状态可追溯 | 所有状态变化有记录可查 |

### 1.2 状态存储位置

```
状态存储层级：
│
├── 迭代SOP索引文档        # 迭代整体状态
│   └── docs/迭代SOP/00_规范/00_迭代SOP_索引.md
│
├── 迭代记录文档          # Sprint级状态
│   └── docs/迭代SOP/07_迭代文档/Sprint-XX/README.md
│
└── 活动状态文件          # 活动级状态（AI维护）
    └── docs/迭代SOP/07_迭代文档/Sprint-XX/状态追踪.md
```

## 2. 状态定义

### 2.1 阶段状态

| 阶段 | 状态值 | 说明 |
|------|--------|------|
| 迭代准备 | 待启动/进行中/已完成 | 准备阶段整体状态 |
| 迭代执行 | 待启动/进行中/已完成 | 执行阶段整体状态 |
| 迭代收尾 | 待启动/进行中/已完成 | 收尾阶段整体状态 |
| 迭代评审 | 待启动/进行中/已完成 | 评审阶段整体状态 |
| 迭代回顾 | 待启动/进行中/已完成 | 回顾阶段整体状态 |

### 2.2 活动状态

| 状态值 | 说明 | 可转换到 |
|--------|------|----------|
| 待启动 | 活动已创建，未开始 | 进行中 |
| 进行中 | 正在执行 | 待审批/待介入/已完成 |
| 待审批 | AI执行完成，等待人类审批 | 通过/进行中 |
| 待介入 | 等待人类响应询问 | 进行中 |
| 通过 | 人类审批通过 | 已完成 |
| 已完成 | 活动完成 | - |
| 暂停 | 活动暂停 | 进行中/已取消 |
| 已取消 | 活动取消 | - |

### 2.3 任务状态（AI任务指令级）

| 状态值 | 说明 | 可转换到 |
|--------|------|----------|
| 待执行 | 任务已下达，未开始 | 执行中 |
| 执行中 | 正在执行 | 待验收/待介入/已完成 |
| 待验收 | 执行完成，等待验收 | 已完成/进行中 |
| 待介入 | 等待人类响应询问 | 执行中 |
| 已完成 | 任务完成 | - |
| 失败 | 执行失败 | 待执行/已取消 |
| 已取消 | 任务取消 | - |

## 5. 依赖管理

### 5.1 依赖类型

| 依赖类型 | 定义 | 处理方式 |
|----------|------|----------|
| 前置依赖 | 必须先完成的任务 | 未完成时阻止启动 |
| 信息依赖 | 需要输入的任务 | 输入未准备好时提示 |
| 资源依赖 | 需要特定资源 | 缺失时提示获取 |
| 审批依赖 | 需要前置审批 | 未审批时阻止执行 |

### 5.2 依赖检查

AI在启动任务前应检查：

```markdown
## 依赖检查报告

### 任务：[任务名称]
### 检查时间：[时间]

#### 前置依赖检查
| 依赖任务 | 状态 | 检查结果 |
|----------|------|----------|
| TASK-001 | 已完成 | ✅ |
| TASK-002 | 进行中 | ⚠️ 建议等待 |

#### 信息依赖检查
| 依赖信息 | 状态 | 检查结果 |
|----------|------|----------|
| 技术方案 | 已确认 | ✅ |
| 接口设计 | 待确认 | ⚠️ 需确认 |

#### 结论
- [ ] 可以启动
- [ ] 建议等待
- [ ] 需补充条件
```

## 6. 上下文传递

### 6.1 活动间上下文传递

活动完成后，应将以下信息传递给后续活动：

| 传递内容 | 传递方式 | 用途 |
|----------|----------|------|
| 技术约束 | 写入技术上下文 | 后续活动参考 |
| 产出文件 | 归档到文档库 | 后续活动引用 |
| 状态更新 | 更新状态追踪文档 | 进度同步 |
| 经验教训 | 记录到本次迭代 | 后续迭代参考 |

### 6.2 上下文引用格式

后续活动引用上下文时，应明确来源：

```markdown
## 上下文引用

### 技术约束（来自技术方案 TP-001）
- 技术栈：Java 17 + Spring Boot 3.x
- 接口规范：RESTful

### 产出引用（来自活动 用户注册API）
- 代码位置：edu-user-service/src/main/java/.../UserController.java
- 数据库表：t_user

### 接口约定（来自活动 接口设计）
- 认证方式：Bearer Token
- 错误码规范：[见API规范文档]
```

## 8. 质量标准

### 8.1 状态准确性

| 标准 | 要求 | 检查方式 |
|------|------|----------|
| 状态更新及时性 | 状态变化后1小时内更新 | 时间检查 |
| 状态准确率 | 状态与实际一致 | 人工抽检 |
| 上下文完整性 | 必要上下文已记录 | 完整性检查 |

### 8.2 上下文质量

| 标准 | 要求 | 检查方式 |
|------|------|----------|
| 上下文可追溯 | 可查到上下文来源 | 追溯检查 |
| 上下文准确性 | 上下文内容正确 | 人工确认 |
| 上下文时效性 | 过期上下文及时清理 | 定期检查 |

## 9. 与其他规范衔接

### 9.1 与活动执行规范的衔接

- 活动状态由AI按照活动执行规范维护
- 活动状态变化触发状态追踪文档更新
- 询问机制产生的待确认事项在状态文档中体现

### 9.2 与询问介入机制的衔接

- 询问产生时添加待确认事项
- 人类响应后更新待确认事项状态
- 统计询问数量用于趋势分析

### 9.3 与迭代流程的衔接

- 阶段转换由人类审批触发
- AI更新阶段状态
- 阶段完成时生成阶段总结

## 11. 上下文文档体系

### 11.1 文档清单

> AI在执行任务时需要读取的上下文文档列表

| 文档 | 位置 | 用途 | 读取优先级 |
|------|------|------|------------|
| **状态追踪文档** | 07_迭代文档/Sprint-XX/状态追踪.md | 迭代/阶段/活动状态 | P0 |
| **技术方案文档** | 07_迭代文档/Sprint-XX/02_技术/技术方案_*.md | 技术栈、API设计、数据库设计 | P1 |
| **需求文档** | 07_迭代文档/Sprint-XX/01_需求/需求文档_*.md | 业务约束、用户故事 | P2 |
| **活动产出文档** | 07_迭代文档/Sprint-XX/03_开发/*.md | 历史产出参考 | P3 |

### 11.2 上下文读取规则

> 详细读取规则见 [06_上下文读取规范](./06_上下文读取规范.md)

| AI角色 | 必须读取的文档 | 说明 |
|--------|----------------|------|
| AI-PM | 状态追踪 + 需求文档 | 获取迭代状态和需求 |
| AI-Architect | 状态追踪 + 技术方案 | 获取迭代状态和技术信息 |
| AI-BE | 状态追踪 + 技术方案 | 获取技术栈和API设计 |
| AI-FE | 状态追踪 + 技术方案 | 获取前端技术和设计 |
| AI-Test | 状态追踪 + 技术方案 + 需求 | 获取测试范围和约束 |
| AI-Reviewer | 状态追踪 + 技术方案 | 获取代码规范和约束 |
| AI-Writer | 状态追踪 + 技术方案 | 获取上下文和产出要求 |
